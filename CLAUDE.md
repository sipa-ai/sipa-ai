# Multi-Agent Assistant — Orchestrator

This project is a configurable multi-agent assistant framework that can be deployed for any organisation. One bot, one router, any number of specialist agents — all managed at runtime via Telegram or the web portal.

## Architecture

### Router Agent

There is always exactly one router agent (`is_router = TRUE` in the `agents` table). It is the single entry point for all Telegram messages. It:
- Handles requests directly (contacts, email, brand, tasks)
- Delegates to specialist agents via the `delegate_to_agent` tool
- Manages the team: creates, updates, and deletes specialist agents

The router agent is seeded automatically on first boot if no router exists. Its name, description, and system prompt can be changed at any time via the portal (`/team`) or via Telegram.

### Specialist Agents

Specialist agents are created dynamically — via Telegram or the portal's `/team/new` form. They live only in the `agents` table (no `.md` files). Each specialist has:
- A `name` (slug, used for delegation routing)
- A `description` (one-liner shown in the router's context)
- A `system_prompt` (generated by Claude from the user's description)
- A `model` (defaults to `claude-sonnet-4-6`)

### Dynamic Routing

The router's system prompt is extended at runtime with an `AVAILABLE AGENTS:` block listing all current specialist agents (name, model, description). This means adding or removing a specialist instantly changes what the router can delegate to — no code change required.

### Tools

The router has a fixed tool set (contacts, email, brand, tasks, posts, team management). Specialist agents created via Telegram receive a default read-only tool set (`get_all_tasks`, `get_all_posts`). Specialists created or edited via the portal can have their instructions customised but tools remain code-defined.

To give a new specialist access to additional tools (e.g. `create_post`, `send_approved_email`), add them to `_AGENT_TOOLS` in `bot.py`.

## Key Files

| File | Purpose |
|------|---------|
| `CLAUDE.md` | This file — architecture overview |
| `context/brand.md` | Brand seed file — loaded into DB on first boot, editable at runtime |
| `bot.py` | Telegram bot, router loop, tool execution, scheduler |
| `db.py` | PostgreSQL schema, seed, all data access functions |
| `portal.py` | FastAPI admin portal |
| `services/llm.py` | Anthropic API wrapper, system prompt builder |

| `services/linkedin.py` | LinkedIn post publishing |
| `services/content.py` | Caption and post generation |
| `services/media.py` | Image/video generation |
| `data/contacts.json` | Seed data for the outreach task pipeline (optional) |

## Setup for a New Organisation

1. Set environment variables (see `app.yaml`)
2. Set brand guidelines — edit `context/brand.md` before first boot, or update via Telegram after boot
3. On first boot, the router agent is created automatically
4. Create specialist agents via Telegram: *"Create a new team member who handles [X]"*
5. Or create them manually in the portal at `/team/new`

## Telegram Agent Management

The router handles these requests naturally in conversation:

- *"Create a new team member who handles social media"* → creates agent, asks for name/model if needed
- *"Update [name]'s instructions to also handle LinkedIn"* → updates system_prompt
- *"Delete [name]"* → confirms then removes
